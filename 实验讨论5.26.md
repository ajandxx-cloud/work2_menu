这条路线和你刚刚上传的方案是一致的：Work 2 不再围绕 SPO 展开，而是从 CNN 网格成本预测升级为“面向服务菜单的结构化表征学习框架”。

\# 一、Work 2 的最终实验定位



你的 Work 2 应该这样定义：



\*\*研究问题：\*\*



在 many-to-one DRT 场景中，当每位乘客到达时，平台不仅要决定每个候选上车点的价格，还要决定向乘客展示哪些有限数量的服务选项。传统 CNN 成本预测方法主要把每个候选点作为独立对象预测，难以刻画候选点之间的替代、互补和空间聚集关系。因此，本文提出 CNN-SetMenuNet，将系统状态的空间网格信息与候选菜单集合的注意力表征结合起来，用于动态服务菜单设计与定价。



这个叙事比“SPO 比 MSE 好”更独立。Work 1 是 \*\*dynamic pricing + decision-focused learning\*\*；Work 2 是 \*\*service menu design + set representation learning\*\*。



学术依据也比较稳。Set Transformer 论文强调很多机器学习任务的输入本质是集合，模型应具备 permutation-invariant 或 permutation-equivariant 特性，并通过 attention 建模集合元素之间的相互作用；这正好对应你的候选接驳点菜单。(\[arXiv]\[1]) 你的 Work 1 代码源头 DSPO 使用的是空间—时间状态编码输入 CNN 来做动态选择与定价，这说明 CNN 路线可以作为基准，但 Work 2 可以合理推进到更适合“候选集合”的 set-attention 表征。(\[arXiv]\[2])

\# 二、核心思想：保留定价系统，替换菜单表征模型



你不要推翻现有系统。最稳的设计是：



> \*\*定价模块不动，MNL 选择模型不动，HGS/Hygese 路径评估不动，只替换“如何预测候选点质量、如何选择菜单”的模型结构。\*\*



也就是说：



Work 1 / 当前代码：



\[

\\text{state grid} \\rightarrow \\text{CNN} \\rightarrow \\hat c\_{tk} \\rightarrow \\text{pricing}

]



Work 2 新模型：



\[

\\text{state grid} \\rightarrow \\text{CNN global encoder}

]



\[

\\text{candidate option set} \\rightarrow \\text{Set-Attention menu encoder}

]



\[

\\text{global state embedding} + \\text{option-set embedding} \\rightarrow \\hat c\_{tk} \\text{ or } s\_{tk}

]



\[

\\text{Top-}L \\text{ menu selection} \\rightarrow \\text{Lambert W pricing} \\rightarrow \\text{MNL passenger choice}

]



这里的关键是：\*\*CNN 负责整体运营状态，Set Transformer 负责候选菜单内部关系。\*\*

\# 五、模型结构应该怎样写



你可以把主模型命名为：



\*\*CNN-SetMenuNet\*\*



结构分成三部分。



\## 1. CNN global state encoder



输入当前系统状态网格 (S\_t)，输出全局状态向量：



\[

z\_t = f\_{\\text{CNN}}(S\_t)

]



这个 (S\_t) 可以继续沿用 Work 1 的状态编码，例如当前已确认订单在时间—空间网格中的分布。



\## 2. Option embedding layer



每个候选选项 (k) 的手工特征 (x\_{tk}) 先过一个 MLP：



\[

e\_{tk} = f\_{\\text{emb}}(x\_{tk})

]



然后拼接全局状态向量：



\[

\\tilde e\_{tk} = \[e\_{tk}, z\_t]

]



得到候选集合：



\[

E\_t = {\\tilde e\_{t1}, \\ldots, \\tilde e\_{tK}}

]



\## 3. Set-attention menu encoder



将候选集合输入 self-attention：



\[

H\_t = f\_{\\text{SetAttn}}(E\_t)

]



每个候选点输出一个上下文化后的表示：



\[

h\_{tk}

]



这里“上下文化”的意思是：候选点 (k) 的表示不仅取决于自己，还取决于其他候选点。比如两个 meeting points 很近，模型可以学到它们功能重复；一个点成本低但步行远，另一个点成本高但吸引力强，模型可以学到二者的替代关系。



\## 4. Output head



第一版建议输出 \*\*边际成本预测\*\*：



\[

\\hat c\_{tk} = f\_{\\text{out}}(h\_{tk})

]



然后用预测成本做菜单选择：



\[

M\_t = \\text{TopL}(-\\hat c\_{tk})

]



也就是选预测成本最低的 (L) 个 meeting points。



第二版可以输出 \*\*menu score\*\*：



\[

s\_{tk} = f\_{\\text{score}}(h\_{tk})

]



然后选 score 最高的 (L) 个 meeting points。



我建议第一版先输出成本，因为和当前代码兼容性最好。

\# 六、训练标签怎样构造



你现在最稳的标签仍然是：



> 每个候选点的真实边际插入成本 (c\_{tk})。



也就是对于每个乘客 (t)，每个候选点 (k)，计算如果该乘客选择 (k)，系统路径成本大概会增加多少。



第一版可以沿用当前代码中的 cheapest insertion / HGS 近似标签，不要马上做完整反事实重优化。因为完整反事实标签虽然更严谨，但计算量会大很多。



标签形式：



\[

\\mathcal{D}

===========



{

(S\_t, X\_t, c\_t)

}

]



其中：



\[

X\_t = {x\_{t1}, \\ldots, x\_{tK}}

]



\[

c\_t = \[c\_{t1}, \\ldots, c\_{tK}]

]



训练目标：



\[

\\mathcal{L}\_{cost}

==================



\\frac{1}{K}

\\sum\_{k=1}^K

\\text{Huber}(\\hat c\_{tk}, c\_{tk})

]



不用 SPO。

不用复杂 decision-focused loss。

主贡献是模型结构。

\# 七、是否加入 ranking loss？



第一版不建议加。先只用 Huber / MSE 训练成本预测。



正式论文版可以加入一个辅助排序损失，但要明确它不是 SPO：



\[

\\mathcal{L}

===========



\\mathcal{L}\*{cost}

\+

\\lambda

\\mathcal{L}\*{rank}

]



其中 (\\mathcal{L}\_{rank}) 可以是 pairwise ranking loss：



\[

\\mathcal{L}\_{rank}

==================



\\sum\_{i,j}

\\max(0, 1 - (s\_i - s\_j)\\cdot \\text{sign}(c\_j - c\_i))

]



直觉是：如果真实成本 (c\_i < c\_j)，模型应该让选项 (i) 的 score 更高。



但是我建议：



\* 第一版：只用 Huber；

\* 第二版：Huber + ranking auxiliary loss；

\* 不要叫 SPO；

\* 不要把贡献点写成 loss；

\* loss 只是辅助。

\# 十、核心指标应该怎样设置

\## 1. 预测与排序指标



这些指标证明你的模型结构确实学得更好：



| 指标                   | 含义                     |

| -------------------- | ---------------------- |

| MAE                  | 成本预测绝对误差               |

4. 是否只是参数更多导致效果好？
5. 是否有实际运营意义？

应该叫：

# 4. 怎样设计才够 TR-E

我建议你把文章贡献写成三层。

## 第一层贡献：提出新的服务菜单决策问题

你不能只说“上车点推荐”。要说：

> 现实 DRT 平台不是向乘客展示所有可能上车点，而是展示一个有限服务菜单。菜单设计会同时影响乘客接受率、上车点选择、车辆路径成本和平台收益。因此，服务菜单不是简单推荐问题，而是一个 choice-based transportation service operations problem。

这就把问题从普通推荐提升到 **运营决策问题**。

你的决策对象是：

[
M_t \subseteq O_t,\quad |M_t|=L
]

其中 (M_t) 是展示菜单，(O_t) 是候选上车点集合。

乘客看到：

[
{(\text{boarding point}, \text{walking distance}, \text{estimated in-vehicle time}, \text{price})}_{k\in M_t}
]

平台目标是：

[
\max \ \mathbb{E}[\text{revenue} - \text{routing cost}]
]

这比“预测成本”强很多。

---

## 第二层贡献：提出适合菜单集合的结构化表征模型

你的方法贡献不是“用了 Transformer”，而是：

> 传统 CNN 只能编码系统空间状态，不能自然处理候选服务选项之间的替代和互补关系。本文提出 CNN-SetMenuNet，将候选上车方案表示为无序集合，并利用 set attention 学习选项间相互作用。

这里要强调三个点：

1. **Permutation equivariance / invariance**：候选上车点没有自然顺序；
2. **Option interaction**：菜单项之间存在替代关系；
3. **Hybrid representation**：CNN 编码全局运营状态，Set encoder 编码候选菜单结构。

这样模型贡献才成立。

---

## 第三层贡献：把菜单学习、定价和路径成本闭环起来

你不能只离线预测菜单。要形成闭环：

[
\text{menu design} \rightarrow \text{pricing} \rightarrow \text{passenger choice} \rightarrow \text{booking set} \rightarrow \text{routing cost}
]

这点非常重要。TR-E 更看重运营系统，而不是单纯 ML 指标。你需要证明：

1. 菜单预测更准；
2. 乘客选择更合理；
3. 运营成本下降；
4. 净利润提高；
5. 退出率没有恶化；
6. 算法实时可运行。

这样才像 TR-E 文章。

# 6. 你当前方案最可能被质疑的地方

如果投稿 TR-E，审稿人可能会质疑下面几点。

## 质疑 1：这是不是只是一个 neural network replacement？

应对方式：

> 不把贡献写成“替换 CNN”，而是写成“服务菜单设计决策框架”。CNN-SetMenuNet 是为了解决候选菜单集合的无序性和选项交互。

## 质疑 2：为什么不是 GNN？

应对方式：

> Work 2 的对象是 passenger-facing candidate service menu，不是固定路网图。候选项集合在每位乘客到达时动态变化，没有稳定图结构；Set-based encoder 更自然。GNN 可以作为鲁棒性对照或扩展，但不是主模型。

## 质疑 3：为什么不是直接优化 routing？

应对方式：

> 乘客订票时最终路线尚未确定。平台需要在在线信息不完全下做菜单和价格决策。本文研究的是前端服务菜单设计如何影响后端 routing cost，而不是替代后端 routing solver。

## 质疑 4：MNL 参数是不是人为设定？

应对方式：

> 做敏感性分析：价格敏感度、步行敏感度、车内时间敏感度、outside option utility 都要变。

## 质疑 5：结果是否只是调参得到？

应对方式：

> 多 seed、多场景、多候选池、多菜单大小、多需求强度、跨实例泛化。结果必须稳。

| 加入强真实案例 + oracle upper bound + 清晰管理洞察 | TR-E 比较有竞争力                        |
| 再加入理论性质或强算法创新                         | TR-E / TR-C 都更稳                    |
