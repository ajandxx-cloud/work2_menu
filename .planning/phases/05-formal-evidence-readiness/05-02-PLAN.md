<objective>
Wire the Phase 5 readiness report into formal artifact generation so
`build_artifacts.py --claim-ready` cannot produce formal claim-ready artifacts
unless a passed readiness JSON matches the source run and all existing row and
artifact gates also pass.
</objective>

<truths>
- Formal readiness JSON is a hard prerequisite for formal `--claim-ready`.
- Readiness passing does not override row-level or artifact-level blockers.
- Diagnostic/status artifacts are allowed for blocked states only when the user
  explicitly allows incomplete output.
- Readiness metadata must flow into `ARTIFACT_STATUS.json`, sidecars, and
  `CLAIM_GUARD.json`.

## Decision Coverage
- D-06: provenance bundle retained in artifact metadata.
- D-08: dependency snapshot path/hash referenced from artifacts.
- D-09: original command and resolved configuration recorded.
- D-13: formal claim-ready requires passed readiness JSON.
- D-14: all formal prerequisites hard-block claim-ready status.
- D-15: blocked readiness may still produce diagnostic/status artifacts.
- D-16: one bad formal row blocks the whole formal run.
- D-17: artifact gates remain authoritative after readiness passes.
</truths>

<tasks>

## Task 1 - Load And Validate Readiness JSON

**Type:** code
**Files:**
- `work2_coding/Src/artifact_status.py`
- `work2_coding/Src/artifact_builder.py`
- `work2_coding/scripts/build_artifacts.py`

**Action:**
1. Add a small readiness loader/validator that accepts a path to
   `FORMAL_READINESS.json`.
2. Extend `build_artifacts()` and the CLI with `--readiness-json`.
3. When `claim_ready=True` and source rows are formal, require the readiness
   JSON.
4. Validate that readiness:
   - exists and parses
   - has `status="passed"`
   - has `claim_ready_allowed=true`
   - has `git_dirty=false`
   - has a dependency snapshot path/hash
   - has checkpoint status `loaded` and checkpoint hash
   - matches the source run manifest hash
   - matches formal checkpoint hash/status in source rows
5. If validation fails, return or raise a clear blocked reason such as
   `formal claim-ready artifacts require passed readiness JSON`.

**Verify:**
```powershell
cd work2_coding
python scripts/test_artifact_gates.py
```

**Acceptance Criteria:**
- Formal `--claim-ready` without readiness JSON is blocked.
- Failed, dirty, stale, or mismatched readiness JSON is blocked.
- Passed matching readiness JSON permits artifact classification to proceed to
  the existing row/artifact gates.
