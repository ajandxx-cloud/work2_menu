<objective>
Add a dedicated formal readiness preflight that checks the formal manifest,
checkpoint, load status, dependency snapshot, git provenance, settings, output
paths, and reproducibility metadata without executing formal replay.
</objective>

<truths>
- Runtime root is `work2_coding/`.
- Formal V1 evidence uses `formal_robust_menu`.
- Formal replay is out of scope for this readiness script.
- Missing or failed formal checkpoint state is a blocked readiness result.
- A passed readiness report is prerequisite evidence, not empirical evidence.
- Do not commit checkpoint binaries or hand-edit generated rows or paper artifacts.

## Decision Coverage
- D-01: formal checkpoint preflight plus generation guidance.
- D-02: checkpoint readiness alone does not unlock claims.
- D-03: checkpoint path, SHA256, load smoke, and loaded row metadata proof.
- D-04: blocked readiness report on checkpoint failure.
- D-05: checkpoint binary remains under runtime output tree.
- D-06: full reproducibility provenance bundle.
- D-07: dirty git hard-blocks formal claim-ready status.
- D-10: dedicated readiness script.
- D-11: JSON and Markdown reports.
- D-12: no formal replay.
</truths>

## Task 1 - Add Formal Readiness Helpers

**Type:** code
**Files:**
- `work2_coding/Src/formal_readiness.py`
- `work2_coding/Src/study_execution.py`
- `work2_coding/Src/artifact_status.py`

**Action:**
1. Create a reusable `formal_readiness.py` module with a public helper such as
   `check_formal_readiness(study="formal_robust_menu", output_root=None, allow_dirty=False)`.
2. Load and validate the formal manifest through existing experiment-contract
   helpers.
3. Resolve the formal shared checkpoint using existing `study_execution`
   checkpoint helpers.
4. If the checkpoint is missing, return a blocked result with:
   - `code="missing_formal_checkpoint"`
   - manifest checkpoint path
   - resolved checkpoint path
   - expected status `loaded`
   - generation command:
     `python scripts/train_shared_checkpoint.py --study formal_robust_menu --checkpoint-path outputs/shared_training/work2_robust_menu/formal/supervised_ml.pt`
5. If the checkpoint exists, compute SHA256 and run a minimal load smoke using
   the same model load path used by actual replay.
6. Build a representative formal row metadata probe or equivalent explicit
   assertion that the loaded checkpoint can produce
   `checkpoint_load_status="loaded"` and the same checkpoint hash.
7. Collect git provenance and hard-block with `code="dirty_git"` when
   `git_dirty=true` unless `allow_dirty=True` is provided for tests only.

**Verify:**
```powershell
cd work2_coding
python scripts/test_formal_readiness.py
```

**Acceptance Criteria:**
- Missing checkpoint readiness writes a blocked result with exact remediation.
- Existing checkpoint readiness records SHA256 and loaded status.
- Dirty git blocks formal claim-ready readiness.
- No formal replay is attempted.

## Task 2 - Persist Dependency Snapshot And Readiness Reports

**Type:** code
**Files:**
- `work2_coding/Src/formal_readiness.py`
- `work2_coding/scripts/check_formal_readiness.py`
- `work2_coding/scripts/test_formal_readiness.py`

**Action:**
1. Persist a dependency snapshot JSON under the selected readiness output
   directory using `collect_environment_provenance(include_freeze=True)`.
2. Compute and record the dependency snapshot SHA256.
3. Write `FORMAL_READINESS.json` with:
   - schema version
   - status: `passed` or `blocked`
   - claim_ready_allowed boolean
   - blockers list
   - study name, tier, run mode
   - manifest path/hash
   - settings hash or resolved-settings hash
   - checkpoint path/hash/load status
   - checkpoint sidecar path/hash when present
   - git provenance
   - dependency snapshot path/hash
   - readiness command and timestamp
4. Write `FORMAL_READINESS.md` summarizing the same gates for human review.
5. Keep file names stable so artifact gates can reference them.

**Verify:**
```powershell
cd work2_coding
python scripts/test_formal_readiness.py
```

**Acceptance Criteria:**
- JSON and Markdown reports are written for both passed and blocked states.
- Dependency snapshot is a separate file with a recorded hash.
- Blocked reports remain useful audit artifacts but do not claim readiness.

## Task 3 - Add Thin CLI

**Type:** code
**Files:**
- `work2_coding/scripts/check_formal_readiness.py`

**Action:**
1. Add an argparse CLI with options:
   - `--study` defaulting to `formal_robust_menu`
   - `--output-root` defaulting to `outputs/formal_readiness`
   - `--allow-dirty` for tests and explicit diagnostics only
2. Print a compact JSON payload with report paths, status, blocker count,
   checkpoint status, dependency snapshot hash, and git dirty status.
3. Return a non-zero process exit code when readiness is blocked unless an
   explicit diagnostic flag is added and documented.

**Verify:**


**Acceptance Criteria:**
- CLI writes both reports.
- CLI fails closed when the real formal checkpoint is missing or git is dirty.
- CLI output includes exact report paths for follow-up artifact commands.
