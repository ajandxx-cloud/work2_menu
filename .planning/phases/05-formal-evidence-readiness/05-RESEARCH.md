# Phase 5 Research: Formal Evidence Readiness

**Date:** 2026-06-14
**Runtime root:** `work2_coding/`
**Status:** Complete

## Research Summary

Phase 5 should add a dedicated formal readiness layer rather than folding more
preflight behavior into `scripts/run_study.py` or relying only on artifact
classification after a run. The current code already has most of the raw
building blocks:

- `work2_coding/Experiments/studies/formal_robust_menu.yaml` declares the
  formal seven-policy mainline family, `menu_k=3`, five paired splits, and a
  required shared checkpoint at
  `outputs/shared_training/work2_robust_menu/formal/supervised_ml.pt`.
- `work2_coding/Src/study_execution.py` can resolve checkpoint paths, hash
  checkpoint files, inspect missing prerequisites, block unsafe formal runs,
  and collect git provenance.
- `work2_coding/scripts/train_shared_checkpoint.py` can generate a deterministic
  shared predictor checkpoint and sidecar, with dirty-git blocking by default.
- `work2_coding/Src/artifact_status.py` classifies formal artifacts and already
  blocks missing dependency snapshots and bad checkpoint status.
- `work2_coding/Src/artifact_builder.py` writes `ARTIFACT_STATUS.json` and
  metadata sidecars, but it currently treats an inline environment snapshot as
  enough for claim-ready formal artifact generation.

## Key Findings

### Checkpoint Readiness

The formal manifest points at a concrete checkpoint target. Discussion context
reported that the target was missing. Phase 5 should make that missing state a
first-class readiness result: `claim_ready=false`, `status=blocked`, a precise
blocker code, and the command needed to create or inspect the checkpoint.

The generation command should reuse the existing checkpoint trainer rather than
inventing another path:

```powershell
cd work2_coding
python scripts/train_shared_checkpoint.py --study formal_robust_menu --checkpoint-path outputs/shared_training/work2_robust_menu/formal/supervised_ml.pt
```

The checkpoint is not committed to git. The readiness report records path,
SHA256 hash, sidecar path/hash when present, load-smoke result, command, and
manifest/settings provenance.

### Readiness Report Shape

The readiness script should perform no formal replay. It should validate that a
formal replay would be auditable and fail closed before expensive execution.
The useful output shape is:

- machine-readable JSON for artifact gates and automation;
- Markdown report for review and manuscript/audit handoff;
- stable blocker codes for missing checkpoint, failed checkpoint load,
  dirty git, invalid manifest contract, missing dependency snapshot, unsafe
  output path, and incomplete provenance.

### Provenance Snapshot

`collect_environment_provenance(include_freeze=True)` already captures Python,
platform, package versions, and `pip freeze`. Phase 5 should persist that
snapshot as a separate file under a formal readiness or run output directory,
hash it, and make artifact claim-ready mode verify the referenced snapshot.
This avoids treating an ephemeral in-memory dependency object as the only source
of provenance.

### Artifact Gate Integration

`build_artifacts.py --claim-ready` should require a passed readiness JSON for
formal runs. The readiness JSON must match the source formal run by manifest
hash, checkpoint hash/status, git commit, and passed status. If it is absent,
failed, dirty, or mismatched, artifact generation should block and still allow
diagnostic/status artifacts only through `--allow-incomplete`.

### Verification Strategy

Phase 5 tests can stay script-style and cheap:

- unit-style tests for missing formal checkpoint and dirty git blocking;
- synthetic readiness-success tests using a temporary checkpoint generated by
  `train_shared_checkpoint.py`;
- artifact-gate tests proving formal `--claim-ready` requires readiness JSON;
- an end-to-end readiness command that produces blocked output when the real
  formal checkpoint is absent.

## Recommended Approach

1. Add reusable formal readiness helpers and a thin
   `scripts/check_formal_readiness.py` CLI.
2. Persist dependency snapshot JSON plus SHA256 and reference it from readiness
   JSON/Markdown.
3. Add artifact-builder enforcement for `--claim-ready` formal runs with a
   required `--readiness-json` or manifest-discoverable readiness path.
4. Extend manuscript claim guard/status metadata to expose readiness path,
   readiness hash, checkpoint hash, and dependency snapshot hash.
5. Verify with script-style tests plus a blocked real-formal readiness run.

## Risks

- A dirty worktree is expected during development, but must hard-block formal
  claim readiness unless an explicit test-only override is used.
- The checkpoint trainer currently uses deterministic synthetic proxy data. A
  generated checkpoint can prove pipeline load/provenance readiness, but not by
  itself empirical superiority.
- Readiness passing must not imply formal claims are supported. Formal replay
  rows and artifact gates still decide claim readiness.
- Artifact gates must compare readiness metadata to the exact formal run; using
  a stale readiness JSON from a different manifest/checkpoint would be unsafe.
