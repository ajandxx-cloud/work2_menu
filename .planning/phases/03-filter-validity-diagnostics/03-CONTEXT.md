# Phase 3: Filter Validity Diagnostics - Context

**Gathered:** 2026-05-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Add quantile errors (P50/P90/P95), filtering-decision diagnostics, and false-negative breakdowns for ETA/IVT filtering. This phase has three components: (A) code changes to compute and persist quantile errors in run_menu_compare.py, (B) per-candidate filtering diagnostics with false-negative breakdown by meeting-point type, and (C) manuscript updates to integrate the new filter-validity evidence.

</domain>

<decisions>
## Implementation Decisions

### Quantile Error Logging
- Modify `run_menu_compare.py` `_aggregate_split_results()` to compute P50, P90, P95 from the existing `displayed_eta_abs_errors` and `displayed_ivt_abs_errors` lists using `np.percentile()`.
- Add the quantile metrics to the split result dict alongside existing means.
- Register the new metrics in `research_pipeline.py` `_METRIC_COLUMNS` so they aggregate correctly.
- Backward-compatible: existing mean/bias metrics remain unchanged.

### False-Negative Breakdown
- Add per-candidate logging in `DSPO_Menu.py` `_build_menu_candidates()`: record each pruned candidate's parcel point ID and the distance band of the meeting point.
- Aggregate distance bands into three categories: near (< 500m), mid (500m-1500m), far (> 1500m).
- Store the breakdown in the split result as `fn_pruning_by_distance_band: {near: X, mid: Y, far: Z}`.
- Extract into a summary table showing FN pruning rate by distance band per filter variant.

### Manuscript Integration
- Update `appendix.tex` filter-validity table to include P50/P90/P95 columns alongside existing MAE/bias.
- Add a new table or paragraph showing false-negative breakdown by distance band.
- Update `results.tex` Section 4.2 (Robust ETA Filtering) interpretation to reference the quantile evidence.
- Be honest about what the quantile diagnostics show — large errors mean the filtering signal is coarse, not necessarily that filtering is harmful.

### Claude's Discretion
Exact quantile thresholds, distance-band boundaries, table formatting, and paragraph wording are at Claude's discretion.

</decisions>

<specifics>
## Specific Ideas

- Round 2 reviewer (Weaknesses section) says: "Filter validity remains under-supported. The new bias diagnostics help, but the reported ETA/IVT errors are large and directional. Without P50/P90/P95 errors, filter-decision confusion matrices, and false-negative breakdowns, the central filtering claim is still vulnerable."
- The actionable fix: "Strengthen filter validation: report error quantiles, realized-feasibility confusion matrices for filtered bundles, and false-negative pruning by meeting-point type or distance band."
- The core bottleneck is `run_menu_compare.py` — error lists exist but are only reduced to means. Computing quantiles is a one-line addition per list.
- FN breakdown requires per-candidate metadata that `DSPO_Menu.py` doesn't currently log — this needs careful implementation to avoid performance regression.

</specifics>
