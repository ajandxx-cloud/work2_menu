# Phase 2: Service Product Contract - Context

**Gathered:** 2026-06-13
**Status:** Ready for planning

<domain>
## Phase Boundary

This phase defines the explicit service product contract for Work2 robust time-window service-menu optimization. It turns the implicit service offer shape into a planning and implementation contract for `j=(m,w,p)`, product modes, time-window modes, menu baseline modes, pricing semantics, normalized-row schema, and formal artifact gates.

This phase should not change algorithm behavior beyond what is needed to expose and validate the contract. Downstream phases may implement algorithms, run studies, and regenerate artifacts using this contract.

</domain>

<decisions>
## Implementation Decisions

### Service Product Definition
- **D-01:** Use a unified product contract that explicitly accounts for home service, meeting-point service, and opt-out. Home and meeting-point offers are service products; opt-out is an explicit choice outcome that must not mutate routes and must not be counted as accepted home pickup.
- **D-02:** Add an independent `ServiceProduct` dataclass for the contract layer instead of only extending `ServiceBundle` or `MenuOffer` metadata. It should represent `j=(m,w,p)` clearly, while adapters connect it to existing `ServiceBundle`, `MenuOffer`, and `ChoiceResult`.
- **D-03:** Define `product_mode` as an incremental feature contract:
  - `m`: location or service point only.
  - `m+w`: location or service point plus pickup time window.
  - `m+w+p`: location or service point plus pickup time window plus price/incentive.
- **D-04:** Define `p` as the passenger-facing price/incentive term that enters MNL utility. System-side fields such as `expected_profit`, `net_profit`, `operational_cost`, and `total_cost` must remain separate from `p`.

### Time-Window Contract
- **D-05:** Define `time_window_mode` by paper semantics, not by raw runtime flags:
  - `no_time_window`: product does not contain `w`, and choice/scoring does not enable window utility.
  - `fixed_window`: product contains a fixed-width time window generated by a fixed rule.
  - `adaptive_window`: product contains a time window whose generation and candidate retention can respond to ETA, uncertainty, chance/interval risk, or equivalent robust signals.
- **D-06:** `fixed_window` uses a fixed width aligned to each candidate's ETA or target-arrival rule. The width is common, but the center may vary by candidate.
- **D-07:** `adaptive_window` may use ETA/risk for both window generation and candidate filtering.
- **D-08:** Time-window utility is gated by `product_mode`: `m` disables time/window-deviation utility, while `m+w` and `m+w+p` enable it.
- **D-09:** `no_filter` is diagnostic only and must not be equated with `no_time_window` or used for formal ranking claims.

### Menu Baseline Contract
- **D-10:** Define `menu_mode` as a structured baseline contract:
  - `no_menu`: single default service product.
  - `fixed_menu`: fixed-rule top-k menu.
  - `random_menu`: reproducible random top-k menu.
  - `optimized_menu`: objective-based menu optimization.
- **D-11:** `no_menu` is a single default home product. Opt-out remains an outside choice outcome, not accepted home service.
- **D-12:** The primary `fixed_menu` baseline is nearest/top-k proximity. `top_k_cheapest` may be retained as supplemental or diagnostic, but it is not the main fixed-menu baseline.
- **D-13:** `random_menu` must draw from the same candidate pool and use the same `menu_k` as paired alternatives, with seeded deterministic randomness derived from the replay setting.
- **D-14:** The primary `optimized_menu` method is `service_guarded_expected_profit`.
- **D-15:** Home service remains available in multi-option menus and does not count against the OOH `menu_k`.
- **D-16:** Main experiments compare multiple `menu_k` values: `{1, 2, 3, 5}`.

### Normalized Row And Failure Contract
- **D-17:** Upgrade study rows to `normalized-row-v2` for Phase 2 contract fields.
- **D-18:** The following fields are mandatory in normalized rows: `product_mode`, `time_window_mode`, `menu_mode`, `pricing_mode`, `method`, `candidate_id`, `study_id`, and `manifest_path`.
- **D-19:** Aggregate rows use `candidate_id="aggregate"`. Real candidate/product IDs belong in per-product traces or tables, not in aggregate rows.
- **D-20:** Define `method` as a canonical composition of `product_mode`, `time_window_mode`, `menu_mode`, and `pricing_mode`; table builders may derive human-readable labels from it.
- **D-21:** Failed policy/split executions still produce normalized rows with `status="failed"`, `error_type`, and `error_message`; batches should continue after a candidate failure.
- **D-22:** Record `net_profit`, `operational_cost`, `total_cost`, and `net_price_revenue` separately. Do not reuse the passenger-facing `p` term as a system profit field.
- **D-23:** Behavior metrics required by the contract are `accepted_count`, `served_count`, `served_rate`, `optout_rate`, `home_share`, `meeting_point_uptake_rate`, `menu_utilization`, and `choice_entropy`.

### Artifact And Claim Gates
- **D-24:** Formal claims must exclude rows that are diagnostic, no-filter-only, failed, placeholder-only, or contract-only.
- **D-25:** Checkpoint load status remains explicit in row metadata and formal gates.

### the agent's Discretion
No decisions were delegated to the agent. User selected all contract choices explicitly.

<code_context>
## Existing Code Insights

### Reusable Assets
- `ServiceBundle` already contains `window_start`, `window_end`, `window_center`, `window_width`, `is_home`, `parcelpoint_id`, and capacity fields. It can back the `m` and `w` parts of `ServiceProduct`.
- `MenuOffer` already carries `ServiceBundle`, `price`, predicted ETA/IVT, walking distance, time deviation, utility, profit, and metadata. It can be the execution-facing offer adapted from `ServiceProduct`.
- `ChoiceResult` already separates `accepted_home`, `accepted_meeting_point`, and `opted_out`, including `route_mutates=False` for opt-out.
- `policy_adapters.py` already provides a clean adapter layer from manifest tags to parser overrides.
- `paired_replay.py` already centralizes row schema validation, manifest/settings hashes, trace identity, and checkpoint metadata.
- `study_execution.py` already supports blocked/incomplete rows and explicit checkpoint metadata.

### Established Patterns
- Keep manifest-driven studies and paired replay as the fairness boundary. New comparison modes should be declared through manifests/adapters rather than ad hoc scripts.
- Keep opt-out accounting separate from accepted home pickup and route mutation.
- Keep no-filter diagnostic rows out of formal ranking claims.
- Add schema fields to centralized row validation and artifact gates before relying on them in paper artifacts.
- Use script-style tests under `work2_coding/scripts/` for contract checks.

### Integration Points
- Add `ServiceProduct` near the existing domain dataclasses, likely in `work2_coding/Environments/OOH/containers.py` or a closely related module, with adapter methods to/from `ServiceBundle` and `MenuOffer`.
- Add parser or manifest-level fields for `product_mode`, `time_window_mode`, `menu_mode`, and normalized `pricing_mode`.
- Extend `policy_adapters.py` with V1 product/menu/time-window comparison families while preserving existing robust-menu tags.
- Upgrade `paired_replay.py` row schema from `normalized-row-v1` to `normalized-row-v2`.
- Extend `study_execution.py` to write failed rows with `error_type` and `error_message`, and to continue batches after candidate failure.
- Add artifact/claim gates that exclude diagnostic, no-filter, failed, placeholder-only, and contract-only rows from formal claims.

</code_context>
