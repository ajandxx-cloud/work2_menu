# Phase 11 Plan: Attention Ablation And Design Fix

**Status:** Planned
**Created:** 2026-06-11
**Mode:** MVP

## Goal

Diagnose the failed pilot attention signal with pre-registered pilot-only ablations, then either select one formal candidate or stop the superiority claim.

## Requirements

- ABLT-01: pilot-only strength, feature, and ETA-variant ablation manifests exist when pilot evidence is weak.
- ABLT-02: varied fields and selection criteria are pre-registered before execution.
- ABLT-03: paired replay fairness is preserved; only declared attention fields vary between compared policies.
- ABLT-04: exactly one formal candidate is selected before formal, or the superiority claim is stopped.

## Work Items

1. Add ablation manifests and suite.
   - `pilot_attention_ablation_strength_high.yaml`
   - `pilot_attention_ablation_eta_feature_focus.yaml`
   - `pilot_attention_ablation_shared_eta_stronger.yaml`
   - `work2_attention_ablation.yaml`

2. Add/extend manifest tests.
   - Ensure manifests load.
   - Ensure checkpoint gates remain required.
   - Ensure low/medium regimes are present.
   - Ensure policy differences are a subset of each manifest's `varied_fields`.

3. Pre-register selection criteria.
   - Candidate must complete with loaded checkpoints and non-placeholder rows.
   - Candidate must have complete pairs.
   - Primary `net_objective_proxy_delta_mean` must be positive.
   - Service constraints must pass.
   - Exactly one candidate may be selected. If none or multiple pass, stop the superiority claim.

4. Run ablations.
   - Execute each ablation manifest.
   - Summarize primary/secondary deltas.

5. Decide.
   - Write `11-ABLATION-RESULTS.md`.
   - Write `11-FORMAL-CANDIDATE-DECISION.md`.

## Acceptance Criteria

- All ablation manifests pass contract tests.
- Ablation results are recorded honestly.
- The final decision is either one selected formal candidate or explicit claim stop.
